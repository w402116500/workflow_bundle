# workflow_bundle v0.5.0 Release Notes

发布日期：2026-04-18

对应 tag：`v0.5.0`

## 概览

`v0.5.0` 是 `workflow_bundle` 首个正式语义化版本。
该版本在 `v0.5.0-rc1` 的基础上完成最终收口，确认版本治理、公共仓库定位和发布识别能力已经稳定，可以作为后续迭代的正式起点。

## 与 `v0.5.0-rc1` 的关系

- 正式版建立在 `v0.5.0-rc1` 验证通过的能力之上。
- 本次收口以版本口径统一、文档归档补齐和正式发布确认为主，不额外引入新的 workflow 功能面。
- `v0.5.0-rc1` 继续保留为首个 semver 预发布快照，便于回看正式版之前的预发布状态。

## 本次重点

### 1. 建立正式版本治理基线

- 根目录 `VERSION` 现作为正式版本单一真源持续生效。
- `CHANGELOG.md` 与 `docs/releases/` 共同形成“结构化变更记录 + 对外发布说明”的双层记录方式。
- `python3 workflow_bundle/tools/cli.py version` 已成为默认版本查询入口，可直接输出当前 semver、tag、commit 与 dirty 状态。

### 2. 固定公共仓库的通用 bundle 定位

- 公开文档继续以通用论文工作流 bundle 为定位，不再混入具体项目快照叙事。
- `workflow/configs/current_project_manifest.json` 与 `workflow/configs/current_workspace.json` 保持为官方示例实例。
- 发布说明目录 `docs/releases/` 已建立，可为后续 semver 版本沿用同一归档方式。

### 3. 让 workflow 产物可追溯到版本

- handoff / resume 已能直接显示当前 bundle version。
- `build_summary.json`、`release_summary.json`、`final_summary.json`、`selftest_summary.json` 会持续记录生成时所用的 bundle 版本信息。
- 对外发布与问题回查时，可以明确关联到具体 workflow 版本，而不再依赖历史聊天上下文或中文 milestone tag 猜测。

## 对使用者的直接影响

- 现在可以把 `v0.5.0` 作为正式稳定版本引用。
- 若要确认当前环境运行的是哪一版 workflow，优先执行 `python3 workflow_bundle/tools/cli.py version`。
- 后续若进入功能增强或修复版本，建议继续沿用 semver tag、`CHANGELOG.md` 与 `docs/releases/` 的组合治理方式。

## 回归结论

正式版收口前已完成以下校验：

- `python3 -m py_compile tools/cli.py tools/core/bundle_version.py tools/core/release_summary.py tools/core/runtime_state.py tools/core/selftest.py`
- `bash workflow/scripts/check_bundle_sync.sh`
- `python3 tools/cli.py version`
- `python3 tools/cli.py selftest`
- `git diff --check`

## 已知边界

- 本版本确认的是 workflow bundle 的正式发布基线，不代表所有后续能力都已冻结。
- GitHub Release 页面可在仓库层另行补充，但不影响 `VERSION`、Git tag 与仓库内 release note 的正式性。
